Chapter 10

Hashing

1 Learning Objectives

  • Understand what a Hash Function and Hash Table are.
  • Understand Collisions and resolution via Chaining vs Open Addressing.
  • Analyse average-case O(1) vs worst-case O(n) complexity and why it matters.
  • Understand Load Factor and Rehashing.
  • Solve classic hashing problems: duplicates, frequency counting, anagram grouping.
  • Use Python dictionaries and sets effectively as hash-based structures.

2 Definitions

Hash Function — a function that converts a key (of any size) into a fixed-size integer (hash code), used to determine the index/bucket in a Hash Table.
Hash Table — a data structure that maps keys to values using a hash function for near-instant (average O(1)) lookup, insert, and delete.
Collision — when two different keys produce the same hash code (or the same table index) — must be resolved so both keys can coexist.

Collision Resolution

ChainingOpen Addressing
Each bucket holds a Linked List of entriesOn collision, probe for the next free slot
Simple, handles high load factor gracefullyBetter cache locality, but clustering risk
Extra memory for list pointersNeeds careful deletion handling (tombstones)

Load Factor & Rehashing

load_factor = number_of_elements / table_size

When the load factor crosses a threshold (commonly 0.7), the table rehashes — doubles in size and reinserts everything. This is O(n) but happens rarely, so insert is still O(1) amortised.

5 Real-Life Examples & Key Points

Phone Book App: name → number, instant O(1) lookup instead of scanning every contact.
Spell Checker: a Hash Set of valid words gives instant "is this word real?" checks.
Caching: URL → cached response, avoiding repeated expensive computation/network calls.
Duplicate Detection: add each item to a Hash Set; if it's already there, you've found a duplicate — O(n) total instead of O(n²).

6 Important Points

  • Hash Table operations are O(1) average case, but can degrade to O(n) worst case with a poor hash function or many collisions.
  • Python's dict and set are both Hash Tables under the hood — keys must be hashable (immutable).
  • Unlike a BST, a Hash Table gives no ordering guarantee — if you need sorted iteration, a Tree is better.
  • A Bloom Filter is a space-efficient alternative when a small false-positive rate is acceptable.

Simple Hash Table with Chaining

class HashTable: def __init__(self, size=10): self.size = size self.table = [[] for _ in range(size)] def _hash(self, key): return hash(key) % self.size def insert(self, key, value): idx = self._hash(key) for pair in self.table[idx]: if pair[0] == key: pair[1] = value; return self.table[idx].append([key, value]) def get(self, key): idx = self._hash(key) for k, v in self.table[idx]: if k == key: return v raise KeyError(key)

Detect Duplicates using a Hash Set

def has_duplicate(arr): seen = set() for x in arr: if x in seen: return True # O(1) average check seen.add(x) return False # overall O(n)

9 Tips & Tricks

"Need O(1) lookup, don't care about order" → Hash Table/Set immediately.
"Count frequency of items" → use a dict or collections.Counter — instant O(n) frequency map.
"Two Sum" style problems almost always optimise from O(n²) to O(n) using a Hash Set/Map.

10 Common Mistakes

Assuming Hash Table operations are always O(1) — worst case (many collisions / bad hash function) is O(n).
Trying to use a mutable type (like a list) as a dictionary key in Python — raises a TypeError since it's unhashable.
Relying on dictionary iteration order for logic that assumes sorting — use a sorted structure if order matters.

        14 Chapter Summary

        A Hash Table maps keys to values via a Hash Function, giving average O(1) insert/lookup/delete. Collisions are resolved via Chaining (linked lists per bucket) or Open Addressing (probing for a free slot). The Load Factor triggers Rehashing to keep performance near O(1). Hash-based structures (dict, set) are the go-to tool for frequency counting, duplicate detection, and O(1) lookups — but sacrifice ordering, unlike Trees.
