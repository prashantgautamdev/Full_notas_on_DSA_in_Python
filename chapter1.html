Introduction to Data Structures & Algorithms

1 Learning Objectives

By the end of this chapter, you will be able to —

  • Explain what a Data Structure and an Algorithm are, and why we study them together.
  • Classify data structures as Linear or Non-Linear, and Primitive or Non-Primitive.
  • Understand Abstract Data Types (ADT) and why they hide implementation detail.
  • Read and reason about Time Complexity & Space Complexity.

2 Definitions

Data Structure — a way of organising, storing and managing data in a computer so that it can be used efficiently. Think of it as the "container" that holds your data in a particular shape.
Algorithm — a finite, step-by-step, well-defined set of instructions to solve a particular problem. It must be unambiguous, take some input, produce an output, and terminate in finite time.
Abstract Data Type (ADT) — a logical/mathematical description of a data structure — what operations it supports (like insert, delete, search) — without specifying how they are implemented internally.

3 Data Structure vs Algorithm

People often mix these up — here's the simple way to remember it:

Data StructureAlgorithm
The container — how data is arrangedThe recipe — steps to solve a problem
Example: Array, Stack, TreeExample: Binary Search, Bubble Sort
Answers: "Where do I keep it?"Answers: "How do I process it?"
Analogy: A kitchen cabinet (data structure) organises your ingredients. A recipe (algorithm) tells you the steps to cook using those ingredients. You need both to make a great dish — just like you need both DS & Algorithms to write efficient programs!

4 Types of Data Structures

  • Primitive: basic built-in types — int, float, char, boolean. Directly operated upon by machine instructions.
  • Non-Primitive: derived from primitive types — Array, Linked List, Stack, Queue, Tree, Graph. Built by the programmer.

6 Important Points

  • Choosing the right data structure can turn an O(n) operation into O(1) or O(log n).
  • Every non-primitive DS can usually be built two ways: using an Array (contiguous memory) or a Linked structure (scattered memory + pointers).
  • ADT = what it does. Data Structure = how it's built. Algorithm = how operations are carried out.
  • A valid algorithm must be finite, definite (unambiguous), and produce correct output for valid input.

7 Time & Space Complexity

Time Complexity — a measure of the amount of time an algorithm takes to run, as a function of input size n. We don't measure in seconds (that depends on hardware!) — we count the number of basic operations.
Space Complexity — a measure of the total memory an algorithm needs, including input storage + extra ("auxiliary") space used during execution.

Best, Worst & Average Case

CaseMeaningExample: Linear Search
Best CaseMinimum time taken (luckiest input)Element found at index 0 → O(1)
Worst CaseMaximum time taken (unluckiest input)Element at last index / not found → O(n)
Average CaseExpected time over all possible inputsElement found somewhere in middle → O(n)
We usually design/analyse for the Worst Case — it gives a safety guarantee: "my program will never be slower than this."

8 Asymptotic Notation

Big-O (O) — Upper bound → "grows no faster than" (worst case)
Big-Omega (Ω) — Lower bound → "grows no slower than" (best case)
Big-Theta (Θ) — Tight bound → "grows exactly at the rate of" (both bounds match)

Growth Rate Comparison (small n → large n)

Thinking O(2n) and O(n) are "different" — constants are dropped! Both are simply O(n). Big-O only cares about growth rate, not exact operation count.
Confusing "Big-O = worst case" as a rule — Big-O is just a notation for an upper bound; it can describe best, average, or worst case depending on what you're analysing.

9 Python Code Examples

Let's see complexity in action 👇

O(1) — Constant Time

# Accessing an array element by index — always 1 step def get_first(arr): return arr[0]

O(n) — Linear Time

# Linear Search — checks every element in worst case def linear_search(arr, target): for i in range(len(arr)): if arr[i] == target: return i return -1

O(log n) — Logarithmic Time

# Binary Search — halves the search space each step def binary_search(arr, target): lo, hi = 0, len(arr) - 1 while lo <= hi: mid = (lo + hi) // 2 if arr[mid] == target: return mid elif arr[mid] < target: lo = mid + 1 else: hi = mid - 1 return -1

O(n²) — Quadratic Time

# Nested loop — every pair is compared def has_duplicate_pair(arr): for i in range(len(arr)): for j in range(i+1, len(arr)): if arr[i] == arr[j]: return True return False

10 Tips & Tricks

Count loops, not lines: 1 loop over n → O(n), 2 nested loops over n → O(n²), loop that halves each time → O(log n).
When comparing complexities, just plug in a big number like n = 1,000,000 mentally — O(log n) ≈ 20 steps, O(n) ≈ 1,000,000 steps. The gap explains why Big-O matters!
If a problem needs frequent search, think Hash Table/BST first. If it needs frequent insert/delete at ends, think Linked List/Deque. If it needs indexed access, think Array.

11 Common Mistakes

Assuming Array and Linked List are "basically the same" — their memory layout (contiguous vs scattered) changes complexity of almost every operation.
Forgetting that space complexity includes auxiliary space (extra variables, recursion stack) — not just the input itself.
Writing pseudocode in strict Python/Java/C++ syntax — pseudocode should be language-agnostic, plain-English steps.

          14 Chapter Summary

          A Data Structure organises data; an Algorithm processes it. Structures are Linear (Array, Stack, Queue, Linked List) or Non-Linear (Tree, Graph). An ADT defines what operations exist without fixing how. We measure efficiency using Time & Space Complexity, expressed via Big-O / Big-Ω / Big-Θ notation, typically analysing the worst case. Choosing the right structure for the right problem is the single biggest lever for writing fast, scalable code.
